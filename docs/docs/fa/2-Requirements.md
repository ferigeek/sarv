# نیازمندی‌ها

## مقدمه

این بخش، از مستنداتی است که در گام «تحلیل و طراحی» نوشته و ارائه می‌شود. هدف از این سند، مشخص‌کردن نیازمندی‌های سیستم، دامنه پروژه، معماری سطح‌بالا، سناریوهای کلیدی، و ابزار‌های مورد استفاده در این پروژه است.

این سند به عنوان بنیان مراحل بعدی توسعه، شامل طراحی دقیق‌تر معماری، پیاده‌سازی سیستم، و توسعه تدریجی قابلیت‌ها مورد استفاده قرار خواهد گرفت.

---
## دامنه پروژه

سیستم هدف، استفاده عمومی خواهد داشت. این شبکه اجتماعی ویژه یک سازمان یا افراد مشخصی نیست؛ هر فرد با هر موضوع و هر هدف، قادر خواهد بود تا از این محصول استفاده کند.

با این حال، دامنه اصلی این پروژه بر طراحی و پیاده‌سازی بخش سمت سرور سیستم متمرکز است. به عبارت دیگر، تمرکز اصلی بر توسعه زیرساخت backend و منطق‌های اصلی سیستم خواهد بود، نه توسعه رابط کاربری یا کلاینت‌های نهایی.

در این پروژه، بخش‌های زیر در دامنه پیاده‌سازی قرار دارند:

- طراحی و پیاده‌سازی API برای ارتباط با کلاینت‌ها و سرویس‌های خارجی
- مدیریت کاربران و داده‌های مرتبط با آن‌ها
- مدیریت و ذخیره‌سازی پست‌ها و محتوای تولید شده توسط کاربران
- پیاده‌سازی سیستم تعاملات کاربران مانند لایک، کامنت و دنبال‌کردن
- طراحی و پیاده‌سازی سیستم feed برای نمایش محتوای کاربران
- پیاده‌سازی ماژول هوشمند (ranking / recommendation system) برای شخصی‌سازی محتوا
- سیستم تحلیل رفتار کاربران و ارائه گزارش از وضعیت سیستم
- طراحی و پیاده‌سازی ساختار پایگاه داده و مدل‌های داده‌ای سیستم

در مقابل، برخی از بخش‌ها به صورت آگاهانه در فاز اولیه این پروژه خارج از دامنه در نظر گرفته شده‌اند:

- توسعه کامل کلاینت (وب یا موبایل) برای کاربران نهایی
- طراحی رابط کاربری گرافیکی (UI/UX)
- استقرار در مقیاس بزرگ و زیرساخت‌های توزیع‌شده پیچیده
- پیاده‌سازی کامل سیستم‌های DevOps در سطح production سازمانی

هر چند که این موارد می‌توانند در مراحل بعدی توسعه سیستم به پروژه اضافه شوند، اما اکنون، تمرکز اصلی بر طراحی یک سیستم backend قابل توسعه، ماژولار و قابل ارتقا است که بتواند در آینده بدون تغییرات اساسی، گسترش پیدا کند.

## سناریوهای کاربردی (Use Cases)

### مرز سیستم

سیستم مورد نظر، یک سرفصل Backend برای شبکه اجتماعی است که مسئولیت مدیریت کاربران، محتوا، تعاملات، تولید فید، رتبه‌بندی هوشمند، تحلیل داده و مانیتورینگ را بر عهده دارد. 
تمام منطق برنامه، پردازش‌های الگوریتمی و ارتباطات میان‌سرویسی (مانند ارتباط جاوا و پایتون) در **داخل مرز سیستم** قرار دارند. کاربران، مدیران و زیرساخت‌های جانبی مستقل، به عنوان **موجودیت‌های بیرونی (Actors)** با این مرز تعامل خواهند داشت.

### بازیگران

در این معماری، بازیگران صرفاً موجودیت‌های خارجی هستند که هدفی را در سیستم آغاز می‌کنند یا سیستم برای تکمیل یک فرآیند به آن‌ها وابسته است:
* **کاربر (User):** بازیگر اصلی سیستم که وظیفه تعامل با پلتفرم (ارسال پست، لایک، دنبال کردن و دریافت فید) را بر عهده دارد.
* **مدیر سیستم (Admin):** بازیگر اصلی سمت مدیریت که مسئول پایش رفتار کلی سیستم، دریافت گزارش‌های کلان و مسدودسازی کاربران خاطی است.
* **سرویس ذخیره‌سازی فایل (File Storage Service):** بازیگر ثانویه (Secondary Actor) و خارجی (مانند یک Object Storage مجزا) که سیستم برای ذخیره و بازیابی رسانه‌ها به آن متصل می‌شود.
* **سیستم مانیتورینگ (Monitoring System):** بازیگر ثانویه و خارجی (مانند Prometheus/Grafana) که متریک‌های حیاتی سیستم را جمع‌آوری و پایش می‌کند.

### دسته‌بندی سناریوها

1.  **مدیریت حساب کاربری (Account Management):** ثبت‌نام، ورود، مدیریت پروفایل.
2.  **محتوا و تعاملات (Content & Interactions):** ایجاد پست، مشاهده پست، ویرایش/حذف، ثبت نظر، لایک/دیس‌لایک، دنبال کردن کاربران.
3.  **فید و هوشمندی (Feed & Intelligence):** دریافت فید زمانی، دریافت فید هوشمند (با زیرفرآیندهای رتبه‌بندی و ثبت لاگ رفتاری).
4.  **مدیریت و تحلیل کلان (Management & Analytics):** مدیریت کاربران توسط ادمین، تحلیل رفتار و تولید گزارش‌های آماری.
5.  **پایش زیرساخت (Infrastructure):** نظارت فنی بر وضعیت سرویس‌ها.

### سناریوهای کلیدی

#### ۱. دریافت فید هوشمند (Get Smart Feed)
* **بازیگران:** کاربر (User)
* **پیش‌نیازها:** کاربر وارد حساب خود شده و توکن معتبر JWT ارائه داده باشد.
* **جریان اصلی:**
    1. کاربر درخواست دریافت فید هوشمند را به API ارسال می‌کند.
    2. سیستم اعتبار توکن JWT کاربر را تأیید می‌کند.
    3. سیستم لیست پست‌های اخیر شبکه را واکشی می‌کند.
    4. سیستم به صورت داخلی لایه هوشمندی (Python) را فراخوانی می‌کند تا پست‌ها را بر اساس الگوریتم (Rank & Personalize) برای این کاربر خاص رتبه‌بندی کند.
    5. هم‌زمان، سیستم یک رکورد رفتاری (Behavior Log) برای این درخواست ثبت می‌کند.
    6. نتایج مرتب‌شده به صورت صفحه‌بندی‌شده (Paginated JSON) به کاربر بازگردانده می‌شود.
* **جریان جایگزین (Fallback):**
    * در صورت از دسترس خارج شدن سرویس هوش مصنوعی پایتون یا بروز خطای محاسباتی، سیستم به صورت خودکار به سناریوی **UC-11 (فید زمانی)** سوییچ کرده و پست‌ها را صرفاً بر اساس زمان انتشار به کاربر نشان می‌دهد تا سرویس‌دهی متوقف نشود (Graceful Degradation).

#### ۲. ایجاد پست جدید (Create New Post)
* **بازیگران:** کاربر (User)، سرویس ذخیره‌سازی فایل (File Storage Service - ثانویه)
* **پیش‌نیازها:** کاربر احراز هویت شده باشد.
* **جریان اصلی:**
    1. کاربر محتوای متنی و فایل رسانه‌ای (در صورت وجود) را ارسال می‌کند.
    2. سیستم فرمت و حجم فایل را اعتبارسنجی می‌کند.
    3. سیستم فایل را به سرویس ذخیره‌سازی خارجی (File Storage) ارسال کرده و آدرس (URL) امن آن را دریافت می‌کند.
    4. اطلاعات متنی پست همراه با URL رسانه در پایگاه داده PostgreSQL ثبت می‌شود.
    5. پیام تایید موفقیت‌آمیز به کاربر نمایش داده می‌شود.
* **جریان جایگزین:**
    * در صورت بروز خطا یا قطع ارتباط با سرویس ذخیره‌سازی فایل، کل تراکنش لغو شده (Rollback)، پست ایجاد نمی‌شود و خطای عدم امکان آپلود رسانه به کاربر اعلام می‌گردد.

#### ۳. تعامل با پست (Like/Dislike Post)
* **بازیگران:** کاربر (User)
* **پیش‌نیازها:** کاربر احراز هویت شده باشد و پست مورد نظر در سیستم وجود داشته باشد.
* **جریان اصلی:**
    1. کاربر دکمه لایک یا دیس‌لایک یک پست را می‌فشارد.
    2. سیستم وضعیت تعامل قبلی کاربر با این پست را بررسی می‌کند.
    3. وضعیت جدید در پایگاه داده ثبت یا به‌روزرسانی می‌شود.
    4. سیستم به صورت غیرهمزمان (Asynchronous) یک سیگنال به فرآیند **UC-15 (ثبت لاگ رفتاری)** ارسال می‌کند تا وزن این تعامل در اولویت‌های بعدی کاربر لحاظ شود.
* **نتیجه:** شمارشگر لایک/دیس‌لایک پست به‌روز شده و تعامل ثبت می‌شود.

#### ۴. تحلیل رفتار کاربران و تولید گزارش (Analyze Behavior & Generate Reports)
* **بازیگران:** مدیر سیستم (Admin)
* **پیش‌نیازها:** داده‌های تعاملی و لاگ‌های رفتاری در طول زمان جمع‌آوری شده باشند.
* **جریان اصلی:**
    1. مدیر سیستم درخواست گزارش آماری (مانند ساعات اوج فعالیت یا پست‌های ترند) را ثبت می‌کند.
    2. سیستم درخواست را به زیرسیستم تحلیل داده (Python Analytics) ارجاع می‌دهد.
    3. سرویس تحلیلی، داده‌های انبوه پایگاه داده را پردازش می‌کند.
    4. گزارش ساختاریافته نهایی تولید شده و در قالب نمودار یا جداول آماری به ادمین نمایش داده می‌شود.

### روابط میان سناریوها

* **رابطه شامل شدن (Include):** سناریوی `Get Smart Feed (UC-12)` همواره شامل اجرای فرآیندهای داخلی `Rank Content (UC-13)` و `Record Log (UC-15)` است.
* **رابطه گسترش (Extend):** سناریوی `Comment on Post (UC-07)` در حقیقت نوعی فرآیند مستقل است که سناریوی پایه‌ای `Create New Post (UC-04)` را گسترش می‌دهد (چرا که کامنت ساختاری دقیقاً مشابه پست دارد اما به یک شناسه والد متصل است).

### سناریوهای بحرانی

سیستم در مواجهه با شرایط زیر باید پایداری خود را حفظ کرده و رفتار پیش‌بینی شده داشته باشد:

-  **از دسترس خارج شدن لایه پایتون (Recommendation Failure):** سیستم باید بدون کرش کردن، فید زمانی ساده (UC-11) را جایگزین کند.
-  **افزایش ناگهانی بار سیستم (Traffic Spike):** در زمان هجوم کاربران، سیستم باید با استفاده از مکانیزم Pagination و لایه‌های Caching بار دیتابیس را کنترل کند.
-  **اختلال پایگاه داده اصلی (Database Outage):** در صورت قطع دیتابیس، مکانیزم‌های پایش (UC-18) باید فوراً به ادمین هشدار داده و سیستم خطای ۵۰۳ (Service Unavailable) استاندارد برگرداند.
-  **کاهش تدریجی عملکرد (Graceful Degradation):** در شرایط فشار حداکثری بر منابع سرور، سیستم مجاز است پردازش‌های سنگین آفلاین (UC-16) را موقتاً متوقف کند تا توان پاسخ‌دهی به درخواست‌های حیاتی کاربران (UC-12 و UC-04) حفظ شود.

## نیازمندی‌های عملکردی

### مدیریت کاربران

سیستم امکان ثبت‌نام، ورود و خروج را برای کاربران فراهم می‌کند. اطلاعات کاربران ذخیره شده و امکان مشاهده و دریافت اطلاعات مربوط به حساب کاربری برای آن‌ها فراهم خواهد بود. همچنین بخشی از اطلاعات و مشخصات حساب کاربری، توسط خود کاربر قابل تغییر است.

علاوه بر این، امکان محدودسازی یا مسدودسازی کاربران برای مدیر سیستم در نظر گرفته می‌شود تا در صورت نیاز بتوان فعالیت کاربران را کنترل یا مدیریت کرد.

### تعاملات کاربران

امکان ارسال پست در اختیار کاربران قرار داده می‌شود. محتوای پست می‌تواند شامل متن، تصویر و ویدیو باشد. کاربران دیگر با مشاهده پست‌ها می‌توانند میزان رضایت یا عدم رضایت خود را نسبت به آن محتوا اعلام کنند. تعداد تعاملات هر پست نیز به صورت عمومی قابل نمایش خواهد بود.

همچنین کاربران می‌توانند نظر خود را درباره هر پست بیان کنند. هر «نظر» خود نوعی پست محسوب می‌شود و ویژگی‌های اصلی هر پست را خواهد داشت؛ با این تفاوت که به یک یا چند پست دیگر وابسته است و به عنوان پاسخ یا ادامه یک گفتگو در نظر گرفته می‌شود.

هر کاربر همینطور می‌تواند کاربران دیگر را دنبال (Follow) کند تا محتوای آن‌ها را بیشتر مشاهده کرده و در feed خود دریافت کند.

تمامی اطلاعات مربوط به تعاملات کاربران توسط سیستم ذخیره و پردازش می‌شود تا در بخش‌های مختلف سیستم، به‌ویژه در سیستم feed و recommendation، مورد استفاده قرار گیرد.

### سیستم Feed

سیستم feed وظیفه تولید و ارائه محتوایی را بر عهده دارد که بیشترین تطابق را با علایق کاربر داشته باشد. این سیستم با استفاده از اطلاعات موجود، تعاملات گذشته کاربران، ارتباط میان کاربران و وضعیت کلی سیستم، تصمیم می‌گیرد چه پست‌هایی و با چه ترتیبی به هر کاربر نمایش داده شوند.

از آنجایی که حجم داده‌ها و تعداد پست‌ها می‌تواند بسیار زیاد باشد، feed به صورت مرحله‌ای و در قالب صفحه‌بندی (pagination) ارائه می‌شود تا کاربر بتواند به تدریج محتوای بیشتری دریافت کند.

همچنین امکان اعمال فیلترهای مختلف برای feed در نظر گرفته می‌شود؛ برای مثال:

- مشاهده پست‌های کاربران دنبال‌شده
- مشاهده پست‌های محبوب یا پرتعامل
- مشاهده جدیدترین محتواها

### سیستم Ranking و Recommendation

این بخش مسئول تحلیل داده‌ها و تعیین میزان ارتباط هر پست با کاربران مختلف است. سیستم با استفاده از اطلاعاتی مانند تعاملات کاربران، میزان محبوبیت پست‌ها، ارتباط میان کاربران، و رفتار گذشته هر کاربر، تلاش می‌کند محتوای مناسب‌تر را در اولویت قرار دهد.

معیارها و الگوریتم‌های مورد استفاده در این بخش می‌توانند در طول توسعه پروژه تغییر کرده یا بهبود پیدا کنند. در نسخه اولیه پروژه، تمرکز بیشتر بر پیاده‌سازی یک ساختار قابل توسعه و آزمایش روش‌های مختلف ranking خواهد بود.

### سیستم آمار و تحلیل کاربران

باشد.

سیستم نظارت و مانیتورینگ، اطلاعات فنی مربوط به عملکرد نرم‌افزار را جمع‌آوری و ثبت می‌کند تا وضعیت سیستم قابل بررسی و تحلیل باشد.

معماری APIها بر پایه REST طراحی می‌شود و مستندات مربوط به endpointها و نحوه استفاده از آن‌ها نیز ارائه خواهد شد.

### مدیریت داده

سیستم باید روابط میان موجودیت‌ها و ساختار داده‌ها را به درستی حفظ کند و با توجه به نوع داده، روش ذخیره‌سازی مناسب انتخاب شود. برای مثال، اطلاعات ساخت‌یافته مانند کاربران و پست‌ها در پایگاه داده رابطه‌ای ذخیره می‌شوند، در حالی که داده‌های حجیم‌تر مانند تصاویر و ویدیوها به صورت جداگانه و با راهکار مناسب مدیریت خواهند شد.

همچنین ساختار ذخیره‌سازی باید به گونه‌ای طراحی شود که قابلیت توسعه و تغییر در آینده را داشته باشد.

## نیازمندی‌های غیر عملکردی

بسیاری از نیازمندی‌های غیرعملکردی، کاملا جدا از هم و دارای راه حل های مجزا نیستند؛ نیازمندی هایی مانند کارایی، مقیاس پذیری، پایداری و امنیت و بسیاری از جنبه های دیگر خروجی تصمیمات درست در ساخت و طراحی نرم افزار هستند. بخش قابل توجهی از این مسائل ریشه در معماری، ابزار مورد استفاده و دانش کافی در پیاده سازی دارند. در حقیقت با رعایت نکردن این موارد در ابتدا، علاوه بر برطرف نکردن این نیازمندی‌ها، جبران و حل این مشکلات در آینده بسیار فرساینده و هزینه بر نیز خواهد بود.

### عملکرد و کارایی

سیستم باید بتواند درخواست‌های کاربران را با زمان پاسخ مناسب و تجربه کاربری روان پردازش کند. از آنجایی که این سیستم شامل عملیات‌هایی مانند مشاهده feed، تعامل با پست‌ها و دریافت داده‌های شخصی‌سازی‌شده است، کارایی مناسب در این بخش‌ها تأثیر مستقیم بر کیفیت تجربه کاربر خواهد داشت.

برای دستیابی به عملکرد مناسب، پردازش داده‌ها باید تا حد امکان بهینه انجام شود و از اجرای عملیات‌های سنگین و غیرضروری در مسیر پاسخ‌دهی مستقیم به کاربر جلوگیری گردد. در مواردی که پردازش‌ها نیاز به زمان بیشتری دارند، می‌توان آن‌ها را به صورت غیرهمزمان یا در قالب فرآیندهای جداگانه انجام داد.

همچنین استفاده از روش‌هایی مانند صفحه‌بندی (pagination)، کاهش حجم داده‌های منتقل‌شده در هر درخواست، و بهینه‌سازی queryهای پایگاه داده، نقش مهمی در کاهش زمان پاسخ‌دهی و افزایش سرعت سیستم دارند.

در بخش‌هایی مانند feed و recommendation، که حجم داده و پیچیدگی پردازش بالاتر است، طراحی الگوریتم‌ها و ساختار داده‌ها باید به گونه‌ای باشد که از افزایش غیرضروری زمان پردازش جلوگیری شود و سیستم بتواند در شرایط بار بالا نیز عملکرد قابل قبولی ارائه دهد.

### مقیاس پذیری

سیستم ساخته شده باید توانایی پاسخگویی به تعداد در حال رشد کاربران، درخواست‌ها و داده‌ها را داشته باشد و بتوان بدون نیاز به تغییرات اساسی در ساختار کلی نرم افزار، ظرفیت و منابع آن را افزایش داد.

از آنجایی که حجم داده‌ها و تعاملات کاربران در شبکه‌های اجتماعی می‌تواند به سرعت افزایش پیدا کند، معماری سیستم باید به گونه‌ای طراحی شود که بخش‌های مختلف نرم افزار تا حد امکان مستقل و ماژولار باشند. این موضوع علاوه بر کاهش پیچیدگی سیستم، امکان توسعه، بهینه‌سازی و ارتقای هر بخش را به صورت جداگانه فراهم می‌کند.

همچنین در بخش‌هایی که حجم پردازش یا تعداد درخواست‌ها بیشتر است، باید امکان استفاده از روش‌هایی مانند pagination، caching، پردازش غیرهمزمان، و بهینه‌سازی queryها وجود داشته باشد تا از ایجاد فشار غیرضروری بر سیستم جلوگیری شود.

ساختار سیستم و انتخاب فناوری‌ها نیز باید به گونه‌ای انجام شود که در آینده، در صورت نیاز، امکان گسترش سرویس‌ها، افزایش منابع پردازشی، یا جداسازی بخش‌های مختلف سیستم با دشواری کم وجود داشته باشد.

### قابلیت توسعه و نگهداری

سیستم باید به گونه‌ای طراحی و پیاده‌سازی شود که در طول زمان، امکان نگهداری، اصلاح و توسعه آن با کمترین پیچیدگی و هزینه ممکن وجود داشته باشد. از آنجایی که این پروژه شامل بخش‌های مختلفی مانند backend، سیستم recommendation و تحلیل داده است، ساختار کلی باید به صورت ماژولار و قابل تفکیک طراحی شود تا هر بخش بتواند به صورت مستقل توسعه یا تغییر داده شود.

تفکیک مسئولیت‌ها (Separation of Concerns) یکی از اصول اصلی در طراحی این سیستم است. هر ماژول باید وظیفه مشخص و محدودی داشته باشد و وابستگی میان بخش‌ها تا حد امکان کاهش یابد. این موضوع باعث می‌شود تغییر در یک بخش، کمترین اثر را بر سایر قسمت‌های سیستم داشته باشد.

همچنین استفاده از ساختار کد استاندارد، نام‌گذاری واضح، و طراحی لایه‌ای (Layered Architecture) به افزایش خوانایی و درک بهتر سیستم کمک می‌کند. این موضوع در کنار مستندسازی مناسب، فرآیند توسعه و همکاری بین بخش‌های مختلف پروژه را ساده‌تر می‌کند.

در بخش‌هایی مانند سیستم ranking و recommendation، باید امکان تغییر و جایگزینی الگوریتم‌ها بدون نیاز به تغییرات گسترده در سایر بخش‌ها وجود داشته باشد. این ویژگی باعث می‌شود سیستم بتواند در طول زمان بهبود پیدا کند و از روش‌های جدیدتر استفاده کند بدون اینکه ساختار کلی آن دچار تغییرات اساسی شود.

در نهایت، استفاده از ابزارهای استاندارد، تست‌پذیری مناسب، و طراحی قابل توسعه، نقش مهمی در کاهش هزینه نگهداری و افزایش عمر مفید سیستم خواهد داشت.

### پایداری و پیوسته در دسترس بودن

سیستم باید تا حد امکان پایدار بوده و بتواند در شرایط مختلف، عملکرد قابل قبولی ارائه دهد. کندی شدید، بروز خطاهای مکرر، یا از دسترس خارج شدن بخش‌های مختلف سیستم، علاوه بر ایجاد تجربه نامناسب برای کاربران، می‌تواند اعتماد و اعتبار نرم افزار را نیز تحت تاثیر قرار دهد.

به همین دلیل، مدیریت مناسب خطاها و جلوگیری از ایجاد اختلال گسترده در سیستم اهمیت بالایی دارد. در صورت بروز مشکل در یک بخش، سایر قسمت‌های سیستم باید تا حد امکان به فعالیت خود ادامه دهند و وابستگی میان اجزای مختلف به گونه‌ای طراحی شود که خرابی یک قسمت باعث از کار افتادن کامل نرم افزار نشود.

همچنین اعتبارسنجی داده‌ها، مدیریت مناسب exceptionها، ثبت رخدادها و نظارت بر وضعیت سیستم، از موارد مهمی هستند که در افزایش پایداری و قابلیت اطمینان نرم افزار نقش دارند.

علاوه بر این، سیستم باید به گونه‌ای طراحی شود که در شرایط افزایش بار یا حجم داده، افت شدید عملکرد یا ناپایداری گسترده ایجاد نشود و امکان بررسی و برطرف کردن مشکلات با پیچیدگی کمتر وجود داشته باشد.

### امنیت و حریم خصوصی

هر چند از امنیت یک نرم افزار نمیتوان به طور کامل اطمینان حاصل کرد، اما میتوان از بسیاری از مشکلات امنیتی رایج و پرتکرار جلوگیری کرد. علاوه بر آن با مدل کردن خطر‌ها و موارد  باارزش در این سیستم،میتوان به درک بهتری از مسیر‌هایی که ممکن است حمله ای صورت بگیرد، دست پیدا کرد و تمرکز بیشتری بر بخش‌های مهمتر گذاشت.

طراحی امن سیستم نیز از آغاز، در برابر بسیار از مشکلات امنیتی پیشگیری می‌کند، و یا هزینه و دشواری برطرف کردن ایرادات را بسیار کاهش میدهد.

اما علاوه بر سعی برای حفاظت از اطلاعات کاربران در مقابل مهاجمین، در ارتباط میان بخش‌های مختلف سیستم، یا در ارتباط با سرویس‌های خارجی، از ارائه و گردش اطلاعات خصوصی و با جزئیات نیز خودداری میشود و حفظ حریم خصوصی، اصلی استوار در توسعه است.

### نظارت و مشاهده‌پذیری

در یک سیستم نرم‌افزاری با چندین بخش مستقل مانند backend اصلی، سیستم recommendation و ماژول‌های تحلیل داده، درک وضعیت داخلی سیستم صرفاً از طریق خطاها یا پیام‌های ساده لاگ کافی نیست. به همین دلیل، مشاهده‌پذیری به عنوان یکی از ویژگی‌های کلیدی سیستم در نظر گرفته می‌شود که امکان تحلیل رفتار سیستم در سطح اجرایی را فراهم می‌کند.

در این سیستم تلاش می‌شود تا اطلاعاتی از وضعیت اجرای سرویس‌ها، الگوی درخواست‌های کاربران و رخدادهای مهم سیستم به صورت ساخت‌یافته جمع‌آوری شود. این اطلاعات باید به گونه‌ای باشند که امکان بررسی عملکرد سیستم در شرایط عادی و همچنین شرایط بار بالا یا خطا را فراهم کنند.

همچنین در بخش‌هایی مانند feed و recommendation که دارای پردازش‌های سنگین‌تر و حساس به داده هستند، مشاهده‌پذیری نقش مهم‌تری در تحلیل عملکرد و شناسایی گلوگاه‌های احتمالی دارد. این موضوع کمک می‌کند تا رفتار سیستم در عمل قابل بررسی بوده و تصمیم‌گیری برای بهینه‌سازی‌ها بر اساس داده‌های واقعی انجام شود، نه صرفاً فرضیات طراحی.

در نهایت، هدف از مشاهده‌پذیری در این سیستم، فراهم‌کردن دید کافی نسبت به وضعیت داخلی اجزای مختلف، تسهیل فرآیند اشکال‌یابی، و افزایش قابلیت اطمینان سیستم در شرایط مختلف است.
### مستندسازی

مستندسازی به صورت دقیق و توضیح واضح برای خواننده، هر چند کاری زمانبر و اغلب دشوار است، اما برای حفظ و نگهداری نرم افزار، فهم کارکرد بخش‌های مختلف برای توسعه قابلیت‌های جدید یا عیب یابی ضروری است.

### سازگاری و قابلیت تعامل

سیستم باید به گونه‌ای طراحی شود که بتواند به راحتی با کلاینت‌ها، سرویس‌ها و ابزارهای مختلف در محیط‌های گوناگون ارتباط برقرار کند. این موضوع شامل سازگاری در سطح ارتباطات شبکه‌ای، فرمت داده‌ها و همچنین قابلیت استفاده در پلتفرم‌های مختلف می‌شود.

ارتباط میان کلاینت‌ها و سرور از طریق APIهای استاندارد و مبتنی بر معماری REST انجام می‌شود و تبادل داده‌ها با فرمت JSON صورت می‌گیرد. این انتخاب باعث می‌شود سیستم بتواند با انواع کلاینت‌ها مانند وب، موبایل یا سرویس‌های خارجی بدون وابستگی به فناوری خاصی ارتباط برقرار کند.

همچنین طراحی APIها باید به گونه‌ای باشد که استفاده از آن‌ها ساده، قابل پیش‌بینی و مستقل از پیاده‌سازی داخلی سیستم باشد تا در صورت تغییر در ساختار داخلی، نیازی به تغییر در کلاینت‌ها وجود نداشته باشد.

در سطح داخلی سیستم نیز تلاش می‌شود تا اجزای مختلف (مانند backend، سرویس هوش مصنوعی و ماژول‌های تحلیل) بتوانند از طریق قراردادهای مشخص و پایدار با یکدیگر تعامل داشته باشند، به طوری که هر بخش بتواند به صورت مستقل توسعه یا جایگزین شود بدون اینکه کل سیستم دچار اختلال شود.

در نهایت، انتخاب فناوری‌ها و ابزارها به گونه‌ای انجام می‌شود که قابلیت اجرا در محیط‌های مختلف (مانند سیستم‌عامل‌های مبتنی بر Linux) حفظ شود و وابستگی سیستم به یک پلتفرم خاص به حداقل برسد.

### یکپارچگی و صحت داده‌ها

یکی از اصول اساسی در طراحی سیستم‌های داده‌محور، حفظ صحت و یکپارچگی اطلاعات در طول زمان است. از آنجایی که در این سیستم داده‌های مختلفی مانند اطلاعات کاربران، پست‌ها، تعاملات و ارتباطات میان کاربران ذخیره و پردازش می‌شود، حفظ هماهنگی میان این داده‌ها اهمیت بالایی دارد.

سیستم باید به گونه‌ای طراحی شود که روابط میان موجودیت‌ها به درستی حفظ شده و از ایجاد داده‌های ناسازگار جلوگیری شود. برای مثال، تعاملات کاربران باید همواره با کاربران و پست‌های معتبر مرتبط باشند و حذف یا تغییر داده‌ها نباید باعث ایجاد وضعیت نامعتبر در سیستم شود.

همچنین در عملیات‌هایی که شامل چند مرحله ذخیره‌سازی یا تغییر همزمان داده‌ها هستند، باید از روش‌هایی استفاده شود که از بروز خطاهای جزئی و ایجاد وضعیت ناقص در داده‌ها جلوگیری کند. این موضوع به حفظ قابل اعتماد بودن سیستم و جلوگیری از بروز خطاهای منطقی در بخش‌های دیگر مانند feed و recommendation کمک می‌کند.

### قابلیت تست و ارزیابی

قابلیت تست یکی از عوامل مهم در تضمین صحت عملکرد سیستم و اطمینان از درستی تغییرات در طول توسعه است. از آنجایی که این سیستم شامل چندین بخش مستقل مانند backend، سیستم recommendation و ماژول‌های تحلیل داده است، امکان تست جداگانه هر بخش و سپس ارزیابی یکپارچگی آن‌ها اهمیت زیادی دارد.

در این سیستم تلاش می‌شود تا بخش‌های مختلف به گونه‌ای طراحی شوند که قابل آزمایش بودن آن‌ها ساده‌تر باشد و بتوان رفتار هر ماژول را به صورت مستقل بررسی کرد. این موضوع باعث می‌شود خطاها سریع‌تر شناسایی شده و تغییرات جدید با اطمینان بیشتری اعمال شوند.

علاوه بر تست‌های واحد و یکپارچه، ارزیابی عملکرد سیستم در شرایط بار بالا نیز اهمیت دارد، به ویژه در بخش‌هایی مانند feed و recommendation که حساسیت بیشتری نسبت به حجم داده دارند. همچنین امکان مقایسه و بررسی کیفیت الگوریتم‌های مختلف در سیستم پیشنهاددهی نیز به عنوان بخشی از فرآیند ارزیابی در نظر گرفته می‌شود.

### تحمل خطا و کاهش تدریجی عملکرد

در سیستم‌های نرم‌افزاری مدرن، به‌ویژه سیستم‌هایی که از چندین سرویس مستقل تشکیل شده‌اند، بروز خطا در یک بخش نباید منجر به از کار افتادن کل سیستم شود. از آنجایی که این پروژه شامل اجزای مختلفی مانند backend اصلی، سیستم recommendation و ماژول‌های تحلیل داده است، طراحی سیستم باید به گونه‌ای باشد که خرابی یا اختلال در یک سرویس، سایر بخش‌ها را تا حد امکان تحت تأثیر قرار ندهد.

در این راستا، اصل «جداسازی خطا» (Fault Isolation) به این معنا است که هر سرویس بتواند به صورت مستقل عمل کند و در صورت بروز مشکل در یک بخش، سایر اجزا همچنان قادر به ادامه فعالیت باشند. برای مثال، در صورت از دسترس خارج شدن سیستم recommendation، سیستم اصلی feed باید همچنان بتواند با استفاده از منطق ساده‌تر (مانند ترتیب زمانی) به کار خود ادامه دهد.

در کنار آن، مفهوم «کاهش تدریجی عملکرد» (Graceful Degradation) بیان می‌کند که در شرایط بروز خطا یا افزایش بار، سیستم به جای توقف کامل، با کاهش کیفیت یا سطح امکانات به ارائه خدمات ادامه دهد. به عنوان نمونه، در صورت فشار بالا بر سیستم، ممکن است برخی قابلیت‌های غیرضروری مانند تحلیل‌های پیشرفته یا شخصی‌سازی عمیق موقتاً غیرفعال شوند، اما عملکردهای اصلی مانند مشاهده پست‌ها و تعاملات همچنان فعال باقی بمانند.

این رویکرد باعث افزایش پایداری کلی سیستم و بهبود تجربه کاربری در شرایط نامطلوب می‌شود، زیرا کاربر حتی در زمان بروز مشکل نیز همچنان می‌تواند از بخش‌های اصلی سیستم استفاده کند.

## ساختار مخزن پروژه

دسته بندی و ساختار پروژه بر اساس نوع وظیفه قسمت‌های مختلف سیستم است. سه بخش کلی این پروژه:

1. منطق اصلی برنامه و مدیریت داده
2. پردازش داده‌ها و امکانات هوشمند برنامه
3. مستندات برنامه

هستند. این بخش‌ها با توجه به مقیاس برنامه در حال حاضر، و راحتی مدیریت به عنوان یک پروژه تک نفره، در مخازن جداگانه‌ای توسعه داده نمی‌شوند، بلکه هر سه در یک مخزن و در directoryهای مختص خود نگه داری می‌شوند.

بنابرین پروژه به طور کلی چنین ساختاری خواهد داشت:

```
social-network-platform/
│
├── backend/
│
├── intelligence/
│   ├── recommendation/
│   └── analytics/
│
├── docs/
|
└── other files and configurations ...
```

## ابزارها و فناوری‌های مورد استفاده

- **Java و Spring Framework:**  
    هسته اصلی سیستم با Java و Spring Framework پیاده‌سازی می‌شود. Java به دلیل کامپایل‌محور بودن، تشخیص خطاها در زمان compile-time و ساختار نسبتاً سخت‌گیرانه، برای توسعه سیستم‌های بزرگ و پیچیده مناسب است. این ویژگی‌ها باعث افزایش پایداری کد و کاهش خطاهای runtime می‌شوند. همچنین وجود اکوسیستم بزرگ و استانداردهای مشخص در توسعه Java، نگهداری و توسعه سیستم را در طول زمان ساده‌تر و قابل پیش‌بینی‌تر می‌کند.
    
    همچنین Spring Frameworkبه عنوان چارچوب اصلی backend استفاده می‌شود، زیرا یک ساختار استاندارد برای طراحی سیستم‌های قابل توسعه و ماژولار فراهم می‌کند. این فریم‌ورک ابزارهای لازم برای ساخت API، مدیریت وابستگی‌ها، کنترل امنیت و ارتباط با دیتابیس را در اختیار قرار می‌دهد و باعث می‌شود معماری سیستم تمیزتر و قابل توسعه‌تر باشد.
    
- **Python:**  
    برای بخش‌های مرتبط با intelligence شامل recommendation و analytics از Python استفاده می‌شود. این زبان به دلیل سادگی، سرعت توسعه بالا و کاربرد گسترده در حوزه‌های data processing و machine learning انتخاب شده است. Python امکان آزمایش سریع ایده‌ها و تغییر الگوریتم‌ها را بدون پیچیدگی زیاد فراهم می‌کند و برای توسعه سیستم‌های مبتنی بر داده بسیار مناسب است.
    
    در این بخش از کتابخانه‌های رایج این حوزه مانند numpy، pandas و scikit-learn استفاده می‌شود که برای پردازش داده، تحلیل و پیاده‌سازی الگوریتم‌های پایه یادگیری ماشین کاربرد دارند.
    
- **PostgreSQL:** 
	پایگاه داده PostgreSQL، به عنوان پایگاه داده اصلی سیستم استفاده می‌شود. این دیتابیس به دلیل پایداری بالا، پشتیبانی از داده‌های ساخت‌یافته و قابلیت اجرای queryهای پیچیده انتخاب شده است. همچنین ویژگی‌هایی مانند transactionهای قوی و عملکرد مناسب در سیستم‌های بزرگ، آن را برای ذخیره‌سازی داده‌های شبکه اجتماعی مناسب می‌کند.
	
- **Markdown و MkDocs:**  
    برای مستندسازی پروژه از Markdown استفاده می‌شود زیرا ساده، سبک و قابل نگهداری است. همچنین MkDocs برای تبدیل مستندات به یک ساختار قابل مشاهده و منظم استفاده می‌شود تا مستندات پروژه به صورت یک وب‌سایت قابل دسترسی باشند. این ترکیب باعث می‌شود مستندسازی هم ساده باشد و هم قابل توسعه.
    
- **Git و GitHub:**  
    برای مدیریت نسخه‌های کد و همکاری تیمی از Git و GitHub استفاده می‌شود. این ابزارها امکان کنترل تغییرات، مدیریت branchها و پیگیری توسعه پروژه را فراهم می‌کنند و برای پروژه‌های در حال رشد ضروری هستند.
    
- **Grafana و Prometheus:**  
    برای مشاهده‌پذیری و مانیتورینگ سیستم از Grafana و Prometheus استفاده می‌شود. Prometheus برای جمع‌آوری و ذخیره metrics سیستم به کار می‌رود و Grafana برای نمایش و تحلیل این داده‌ها استفاده می‌شود. این ترکیب کمک می‌کند وضعیت عملکرد سیستم، مصرف منابع و مشکلات احتمالی به شکل قابل مشاهده و قابل تحلیل در دسترس باشد.
