# طراحی پایگاه داده

## نمای کلی

این پروژه از PostgreSQL به عنوان پایگاه داده اصلی برای یک پلتفرم شبکه اجتماعی استفاده می‌کند.  
این پایگاه داده برای مدیریت کاربران، پست‌ها، تعاملات، رسانه‌ها و ثبت رویدادها جهت تحلیل و یادگیری ماشین طراحی شده است.

---
## نمودار ER (ERD)

نمودار زیر ساختار کامل پایگاه داده شامل موجودیت‌ها، ارتباط‌ها و محدودیت‌ها را نشان می‌دهد.

## موجودیت‌های اصلی

### کاربران (Users)
اطلاعات حساب کاربری و پروفایل کاربران را نگهداری می‌کند.

فیلدهای مهم:
- id (کلید اصلی)
- username (یونیک)
- email (یونیک)
- password_hash
- status (active, suspended, banned, deleted)
- gender (enum)
- profile_picture (ارجاع به media)

### پست‌ها (Posts)
نمایانگر محتوای تولید شده توسط کاربران است.

انواع پست:
- پست معمولی (NORMAL)
- کامنت (COMMENT) از طریق parent_id
- بازنشر (REPOST) از طریق repost_of_id
- نقل‌قول (QUOTE) همراه با ارجاع به یک پست دیگر

هر پست متعلق به یک کاربر است.

### واکنش‌ها (Reactions)
واکنش کاربران به پست‌ها را ذخیره می‌کند.

- reaction_type:
  - 1 = لایک
  - -1 = دیسلایک
- هر کاربر فقط یک واکنش برای هر پست می‌تواند ثبت کند (unique constraint)

### دنبال‌کردن (Follows)
ارتباط دنبال‌کننده و دنبال‌شونده را نگهداری می‌کند.

- follower_id → کاربر دنبال‌کننده
- followed_id → کاربر دنبال‌شونده
- جلوگیری از ثبت تکراری رابطه

### رسانه‌ها (Media)
اطلاعات فایل‌های آپلود شده را نگهداری می‌کند.

شامل:
- اندازه فایل
- نوع MIME
- هش SHA-256 (برای جلوگیری از تکرار فایل)
- مالک فایل

### لاگ رویدادها (Event Logs)
برای ثبت رفتار کاربران جهت تحلیل و یادگیری ماشین استفاده می‌شود.

کاربردها:
- رتبه‌بندی فید
- تحلیل رفتار کاربران
- سیستم پیشنهاددهی

نمونه رویدادها:
- LOGIN
- VIEW_POST
- LIKE_POST
- CREATE_POST
- FOLLOW_USER
- REQUEST_FEED

شناسه کاربر می‌تواند NULL باشد (برای ناشناس‌سازی داده‌ها).

## روابط بین جداول

- هر کاربر می‌تواند چندین پست ایجاد کند
- هر کاربر می‌تواند چندین دنبال‌کننده و دنبال‌شونده داشته باشد
- هر کاربر می‌تواند چندین واکنش ثبت کند
- هر پست می‌تواند چندین واکنش داشته باشد
- پست‌ها می‌توانند کامنت داشته باشند (ارتباط بازگشتی)
- پست‌ها می‌توانند نقل‌قول از پست دیگر باشند (QUOTE)
- هر فایل رسانه‌ای می‌تواند متعلق به یک کاربر باشد
- رویدادها می‌توانند به کاربر، پست یا کاربر هدف مرتبط باشند

---

## استراتژی حذف داده‌ها

- کاربران: حذف کامل پست‌ها، دنبال‌ها و واکنش‌ها (CASCADE)
- پست‌ها: در صورت حذف کاربر حذف می‌شوند
- رسانه‌ها: مالک حذف شود → مقدار NULL می‌شود
- لاگ‌ها: کاربر حذف شود → مقدار user_id برابر NULL می‌شود (برای حفظ داده‌های تحلیلی)

## ایندکس‌ها (برنامه آینده)

برای بهبود عملکرد، ایندکس‌ها روی موارد زیر پیشنهاد می‌شوند:

- posts.user_id
- reactions.post_id
- reactions.user_id
- follows.follower_id
- follows.followed_id
- event_logs.user_id

## نکات

- لاگ‌های رویداد برای سیستم یادگیری ماشین و پیشنهاددهی استفاده می‌شوند
- حذف نرم (soft delete) از طریق deleted_at در پست‌ها و رسانه‌ها پشتیبانی می‌شود
- طراحی پایگاه داده برای بار خواندن بالا (feed system) بهینه شده است
